# Contributing to YDB SLO Action

Welcome! This guide covers development setup, architecture, and conventions for contributors.

## Prerequisites

- **Bun** (package manager): [Install Bun](https://bun.sh)
- **Docker** (for local testing)
- Basic understanding of TypeScript and GitHub Actions

## Setup

```bash
git clone https://github.com/ydb-platform/ydb-slo-action.git
cd ydb-slo-action
bun install
```

## Development Workflow

1. **Make your changes** in `init/`, `report/`, or `shared/` directories
2. **Build the action** to verify everything works:
   ```bash
   bun run bundle
   ```
3. **Commit your changes** — husky will automatically:
   - Run linting and formatting
   - Rebuild `dist/` directory
   - Stage the rebuilt files

**Important:** Never edit files in `dist/` manually! They're auto-generated by the bundler. GitHub Actions can only run JavaScript, not TypeScript, so we bundle TypeScript into optimized JavaScript in `dist/`.

## Testing Locally

You can test the infrastructure locally. Docker Compose uses profiles to control which services start:

| Profile             | Services                                        |
| ------------------- | ----------------------------------------------- |
| _(none)_            | YDB cluster only (1 storage + 5 database nodes) |
| `telemetry`         | Prometheus + Grafana                            |
| `chaos`             | Chaos monkey + blackhole node                   |
| `workload-current`  | Current workload container                      |
| `workload-baseline` | Baseline workload container                     |

```bash
cd deploy

# YDB cluster only
docker compose up -d

# YDB + Prometheus + Grafana + chaos monkey
docker compose --profile telemetry --profile chaos up -d

# Everything (including workloads)
docker compose --profile telemetry --profile chaos --profile workload-current --profile workload-baseline up -d
```

Stop everything with:

```bash
docker compose down
```

### Using Custom Workloads

The `deploy/compose.yml` supports workload containers with configurable environment variables:

```bash
# Run with default settings
docker compose --profile workload-current up

# Override workload duration (in seconds, 0 = unlimited)
WORKLOAD_DURATION=300 docker compose --profile workload-current up

# Use custom workload image
WORKLOAD_CURRENT_IMAGE=my-custom-workload docker compose --profile workload-current up

# Override workload command/arguments
WORKLOAD_CURRENT_COMMAND="--threads 10 --duration 300" docker compose --profile workload-current up

# Combine multiple options
WORKLOAD_CURRENT_IMAGE=my-workload \
WORKLOAD_CURRENT_COMMAND="--verbose --mode stress" \
WORKLOAD_DURATION=300 \
docker compose --profile workload-current up

# Run both current and baseline workloads simultaneously
WORKLOAD_CURRENT_IMAGE=my-workload:pr-123 \
WORKLOAD_BASELINE_IMAGE=my-workload:main \
WORKLOAD_CURRENT_COMMAND="--threads 20" \
WORKLOAD_BASELINE_COMMAND="--threads 10" \
docker compose --profile telemetry --profile workloads up
```

Or create a `.env` file in the `deploy/` directory:

```bash
cp env.example .env
vim .env
```

**Available environment variables:**

| Variable                    | Default                 | Description                                                  |
| --------------------------- | ----------------------- | ------------------------------------------------------------ |
| `WORKLOAD_DURATION`         | `0`                     | Workload duration in seconds (0 = unlimited)                 |
| `WORKLOAD_CURRENT_IMAGE`    | `ydb-workload-current`  | Docker image for current workload                            |
| `WORKLOAD_BASELINE_IMAGE`   | `ydb-workload-baseline` | Docker image for baseline workload                           |
| `WORKLOAD_CURRENT_COMMAND`  | _(empty)_               | Override command for current workload (e.g., `--threads 10`) |
| `WORKLOAD_BASELINE_COMMAND` | _(empty)_               | Override command for baseline workload                       |

**Note:** YDB connection and Prometheus endpoints are pre-configured in `compose.yml` and should not be overridden. See `deploy/env.example` for complete configuration options.

## Code Style

We use automated formatting, so you don't need to worry about style. Just follow these conventions:

- **Import with `.js` extensions**: `import { x } from './module.js'` (ESM requirement)
- **Use `node:` prefix**: `import * as fs from 'node:fs'`
- **Prefer `let` over `const`** (project convention)

Run linting and formatting manually:

```bash
bun run lint    # Fix linting issues
bun run format  # Format code
```

## Commit Message Format

We use emoji-based commit messages for easy scanning:

```
✨ Add custom metrics support

Users can now provide custom Prometheus queries via the metrics_yaml
input parameter. This allows tracking SDK-specific metrics without
forking the action.
```

**Emoji guide:**

- ✨ New feature
- 🐛 Bug fix
- 📝 Documentation
- ♻️ Refactoring
- 🔧 Configuration/build changes
- 🐳 Docker-related changes
- 🧪 Tests
- 🚀 CI/CD changes

**Rules:**

- Use imperative mood ("Add" not "Added")
- Capitalize after emoji
- No period at end of subject line
- Explain WHAT and WHY in the body (not HOW)

## Architecture Overview

### Project Structure

```
init/
  ├── main.ts              # Entry point (deploys infrastructure)
  ├── post.ts              # Cleanup (collects metrics, uploads artifacts)
  └── lib/                 # Utility modules (docker, prometheus, github, etc.)

report/
  ├── main.ts              # Entry point (generates and posts report)
  └── lib/                 # Utility modules (artifacts, html, comment, etc.)

shared/
  ├── metrics.ts           # Metric config parsing (YAML)
  ├── analysis.ts          # Paired-comparison analysis engine
  ├── stats.ts             # Statistical functions
  ├── thresholds.ts        # Threshold config + evaluation
  └── ...

deploy/
  ├── compose.yml          # Docker Compose definition
  ├── metrics.yaml         # Default Prometheus queries
  ├── thresholds.yaml      # Default SLO thresholds
  ├── ydb/
  │   ├── Dockerfile       # YDB node image
  │   └── rootfs/          # Files copied to container root (/)
  ├── chaos/
  │   ├── Dockerfile       # Chaos monkey image
  │   └── rootfs/          # Files copied to container root (/)
  └── app/
      └── ...              # Example workload

dist/                      # Auto-generated (don't edit!)
```

### Design Principles

1. **Separation of Concerns** — Lifecycle files (main.ts, post.ts) orchestrate; utility modules (lib/) do the heavy lifting
2. **Infrastructure as Code** — Docker Compose for services, YAML for metrics, shell scripts for chaos
3. **Artifact-Based Communication** — `init` uploads metrics as GitHub Artifacts, `report` downloads them. This decouples the actions
4. **Configuration Over Code** — Users customize via inputs and config files, not code changes

### GitHub Actions Lifecycle

The `init` action uses GitHub Actions' main/post pattern:

1. `main.ts` runs **before** user steps — deploys infrastructure, starts workloads
2. User's workflow steps run (if any)
3. `post.ts` runs **after** (even if steps fail) — collects metrics, tears down infrastructure

State flows from `main.ts` to `post.ts` using `saveState()`/`getState()` APIs.

### Docker Image Structure

We use the **rootfs pattern** for organizing Docker images (inspired by [Bitnami containers](https://github.com/bitnami/containers)):

1. Each service directory (e.g., `ydb/`, `chaos/`) contains:
   - `Dockerfile` — image definition
   - `rootfs/` — directory structure as it will appear in the container

2. In the Dockerfile, `COPY rootfs /` copies the entire `rootfs/` content to the container's root filesystem

3. Example: `deploy/chaos/rootfs/opt/ydb.tech/scripts/chaos/libchaos.sh` becomes `/opt/ydb.tech/scripts/chaos/libchaos.sh` in the container

### Metrics Collection

1. Metrics are defined as YAML (name, PromQL query, step) in `deploy/metrics.yaml`
2. `init/post.ts` queries the Prometheus API for each metric
3. Results are serialized as JSONL (one JSON object per line) — easier to append and less memory-intensive than JSON arrays
4. Uploaded as GitHub Artifacts: `{workload}-metrics.jsonl`

### Report Generation

1. Downloads artifacts from the current workflow run
2. For each workload: aligns current/baseline time series by timestamp, computes pointwise ratios
3. Evaluates metrics against thresholds (absolute bounds + relative change)
4. Renders HTML report with charts, uploads as artifact
5. Posts summary as PR comment

## Writing Chaos Scenarios

Chaos scenarios are shell scripts in `deploy/chaos/rootfs/opt/ydb.tech/chaos/scenarios/`.

### Template

```bash
#!/bin/sh
set -e

. /opt/ydb.tech/scripts/chaos/libchaos.sh

echo "Scenario: Your description"

# Select a random target
node=$(get_random_database_node)

# Inject fault (sets chaos_active=1 metric)
chaos_inject "my-fault" "${node}"

# Your chaos logic
docker stop "${node}" -t 30
sleep 5
docker start "${node}"

# Wait for recovery
wait_container_healthy "${node}"

# Recover (sets chaos_active=0 metric)
chaos_recover "my-fault" "${node}" "Node recovered"
```

**Naming convention:** `NN-descriptive-name.sh` (e.g., `01-graceful-stop.sh`)

**Helper functions:**

- `get_random_database_node` — random database node container
- `get_random_storage_node` — random storage node container
- `chaos_inject "fault" "node"` — mark fault as active (sends OTLP gauge)
- `chaos_recover "fault" "node" "desc"` — mark fault as inactive
- `wait_container_healthy "container" [timeout]` — wait for container health check

**Golden rules:**

1. **Always restore to healthy state** — don't leave the system broken
2. **Use randomization** — avoid predictable patterns
3. **Use `chaos_inject`/`chaos_recover`** — enables visualization in Grafana

### Existing Scenarios

- `01-graceful-stop.sh` — stops a node gracefully, then restarts
- `02-instant-restart.sh` — immediate docker restart
- `03-sigkill.sh` — SIGKILL termination
- `04-pause-unpause.sh` — container pause/unpause (simulates freezes)
- `05-rolling-restart.sh` — sequential node restarts
- `06-ip-blackhole.sh` — network black hole

## Important Gotchas

### Import Extensions

TypeScript ESM requires `.js` extensions in import paths, even though files are `.ts`:

```typescript
// Correct
import { func } from './module.js'

// Wrong (will fail at runtime)
import { func } from './module'
```

### Docker Compose Working Directory

Always run Docker Compose commands with `cwd` set to the directory containing `compose.yml`. Docker resolves relative paths based on working directory.

### Artifact Naming

Use the pattern `{workload}-{type}.{extension}`:

- `my-workload-metrics.jsonl`
- `my-workload-logs.txt`
- `my-workload-alerts.jsonl`
- `my-workload-metadata.json`

This prevents conflicts when multiple workloads run in the same workflow.
